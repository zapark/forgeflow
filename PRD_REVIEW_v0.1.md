# ForgeFlow PRD v0.1 审查报告（可行性 / 问题 / 改进建议）

基于 `PRD_v0.1.md`，本报告从“战略可行性、技术可行性、交付可行性、商业可行性”四个维度评审，并给出可落地改进方案。

## 1. 总体结论

**结论：方向正确，但当前 PRD 偏“愿景型”，尚不足以直接指导 MVP 开发。**

- 优点：产品定位清晰（Local-First + 多角色协作 + 长周期运行），差异化明确。
- 风险：范围过大、关键指标缺失、技术选型与实现路径未收敛。
- 建议：将 V1.0 拆成“可验证 MVP（8~12 周）+ 扩展里程碑（后续季度）”，先验证核心闭环：
  1) 任务创建 → 2) 可观测执行 → 3) 可恢复调度 → 4) 可审计交付。

---

## 2. 可行性判断

### 2.1 战略可行性：**高**

- 痛点真实：用户确实需要“持续执行、可回放、可控权限”的 AI 自动化系统。
- 市场窗口：Agent 类产品正在从“聊天工具”走向“执行系统”，PRD 方向契合趋势。

### 2.2 技术可行性：**中**

- 可行部分：FastAPI + LangGraph + APScheduler + SQLite/Chroma 的组合可构建单机 MVP。
- 难点部分：
  - “自动安装工具 + 安全沙箱 + 低确认执行”同时成立难度高；
  - Desktop 采用 Electron + WebView、与服务端 C/S 分离后，客户端与服务端可独立打包发布，此项工程复杂度可控，不应作为核心阻塞；
  - 多角色并行下的状态一致性与幂等补偿未定义。

### 2.3 交付可行性：**中低**

- 当前 V1.0 必须实现项达 7 大系统，覆盖编排、调度、安全、审计、生态，超出典型初版承载。
- 缺乏版本边界与验收标准，团队很容易陷入“功能都在做、核心价值未验证”。

### 2.4 商业可行性：**中**

- 分层定价逻辑合理，但“免费版与专业版能力边界”不够可执行。
- 未定义关键转化路径（如：首个自动化任务成功率、7 日留存、任务复用率）。

---


### 2.5 Desktop 架构可行性补充：**中高**

- 采用 Electron 架构并以 WebView 承载主交互层，适合快速复用 Web 前端能力与插件化扩展。
- 架构建议明确为 **C/S 分离**：Desktop 客户端与后端服务独立部署、独立版本管理。
- 发布策略建议：客户端与服务端**分开打包**，通过 API/协议契约保证兼容性；无需强行做一体化打包。
- 结论：Desktop 与服务端分包不是主要风险点，主要风险仍在“协作治理模型与可控自治执行闭环”。

---

## 3. 关键问题清单（按优先级）

## P0（必须先解决）

1. **MVP 范围过宽，缺少“最小可验证闭环”**
   - 风险：延期、质量不稳、价值不清。
2. **缺少量化成功指标（KPI/SLO）**
   - 如任务成功率、平均恢复时长、人工介入率、单任务成本。
3. **安全模型仅描述原则，未给执行细则**
   - 如权限申请流程、审批策略模板、审计字段规范。
4. **Workflow 语义不完整**
   - 节点输入输出契约、失败策略、幂等键、补偿机制未定义。

## P1（影响可扩展）

5. **多角色协作协议不明确**
   - 消息结构、事件顺序、冲突仲裁缺失。
6. **Skill 生命周期不完整**
   - 发布、签名、信任源、回滚与兼容性矩阵未定义。
7. **运维观测能力缺失**
   - 缺少日志分级、Tracing、指标面板、告警策略。

## P2（影响商业化效率）

8. **版本分层未映射到可计费能力单元**。
9. **企业版需求未形成交付规格**（SSO、RBAC、审计导出、离线升级等）。

## 4. 改进建议（可直接纳入下一版 PRD）

### 4.1 重构版本策略

- **V0（2~3 周）**：单角色 + 单工作流模板 + 手动触发 + 日志可见。
- **V1-MVP（8~12 周）**：
  - NL 任务创建（限定模板域）
  - 角色协作（Planner + Operator + Reviewer）
  - Workflow（顺序/并行/重试/人工检查点）
  - Scheduler（Cron + 失败重试）
  - Replay（事件级回放）
- **V1.1+**：Skill Marketplace、企业权限增强、多 Worker 分布式。

### 4.2 补齐验收指标（示例）

- 任务端到端成功率 ≥ 80%（限定 3 类模板任务）
- 失败自动恢复成功率 ≥ 60%
- 平均人工介入次数 ≤ 1 次/任务
- 单任务执行成本（Token+Runtime）可视化覆盖率 100%
- P95 任务启动时延 ≤ 10 秒

### 4.3 明确技术契约

- 定义 `WorkflowSpec v1`：节点类型、I/O Schema、重试策略、超时策略、幂等键。
- 定义 `ToolSpec v1`：权限声明、资源限制、审计字段、错误码。
- 定义 `EventSpec v1`：事件名、payload、trace_id、因果链路。

### 4.4 安全落地建议

- 默认 deny-all，按 Skill 最小权限授权。
- 引入“权限策略模板”（本地文件、网络白名单、命令白名单）。
- 高风险操作（shell/network write）必须 checkpoint。
- 审计日志不可变（append-only）并支持导出。

### 4.5 商业化建议

- 免费版限制“并发任务数、调度频率、可安装 Skill 数量”。
- 专业版提供“高级调度 + 审计 + 团队共享”。
- 企业版提供“SSO/SAML、细粒度 RBAC、私有模型网关、合规导出”。

---

## 5. 建议新增的 PRD 章节

1. **非功能性需求（NFR）**：性能、可靠性、可维护性、安全、可观测性。
2. **边界与不做清单**：明确 V1 不支持的场景。
3. **失败设计**：超时、限流、依赖失败、任务中断时的降级路径。
4. **数据治理**：数据保留、清理策略、隐私分级。
5. **验收用例**：每个核心模块至少 3 个“通过/失败”样例。

---

## 6. 建议的下一步执行顺序

1. 一周内输出《MVP 范围冻结版 PRD v0.2》与指标表。
2. 同步产出三份技术契约：WorkflowSpec / ToolSpec / EventSpec。
3. 用 2 个真实场景做“端到端试运行”：
   - 每日行业信息跟踪
   - 自动化周报生成
4. 基于试运行结果再决定 Skill Marketplace 与企业版优先级。

---

## 7. 一句话结论

ForgeFlow 的产品方向具备较强潜力，但要从“概念完整”走向“交付成功”，关键是**缩 MVP、立指标、补契约、先跑通闭环**。

## 8. 面向竞品的创新定位（避免重复造轮子）

核心原则：**能力创新以“组织协作思想创新”为主，而非底层技术堆栈创新。**

### 8.1 与竞品拉开差异的“思想层创新”

1. **从“单 Agent”升级到“数字化公司”**
   - 不是回答问题，而是完成“需求→经营动作→交付”的闭环。
   - 用户角色是老板，系统角色是可管理、可追责、可复盘的数字化团队。

2. **全智化执行 + 全程可治理**
   - 目标不是“自动化越多越好”，而是“自动化与可控性并存”。
   - 必须同时支持：**可监督、可干预、可控制、可交互、无值守**。

3. **默认复用成熟能力，不重复造轮子**
   - 工作流、调度、模型网关、权限、向量检索优先采用成熟组件。
   - ForgeFlow 的核心价值放在：任务治理模型、角色协作协议、交付标准化。

### 8.2 产品能力重心建议（按优先级）

- **P0：组织协作内核**
  - 目标解析（Goal → Plan → KPI）
  - 角色协作协议（责任边界、交接标准、升级机制）
  - 交付物定义（不同人群的可消费输出模板）

- **P1：过程治理内核**
  - 监督视图（实时状态、风险提示、成本/质量看板）
  - 干预机制（暂停/继续/改道/接管）
  - 控制机制（权限闸门、预算闸门、风险策略）

- **P2：生态扩展内核**
  - Skill 复用市场（内部模板优先）
  - 场景包（开发者/普通用户/企业用户）

### 8.3 三类用户统一架构（同底座、不同体验）

1. **普通用户（老板模式）**
   - 输入目标，系统自动拆解并执行；仅在风险节点提醒。
   - 交付物默认“可直接使用”（报告、邮件草稿、看板）。

2. **开发者用户（工程模式）**
   - 可查看/编辑 Workflow、工具链、策略规则。
   - 支持调试、回放、压测与版本管理。

3. **企业用户（治理模式）**
   - 支持组织权限、审批链、审计导出、私有模型治理。
   - 可设定部门级策略与合规边界。

> 关键设计：三类用户不是三套系统，而是**同一执行内核 + 分层控制面板**。

### 8.4 建议新增“全智化执行标准”（写入 PRD）

定义任务生命周期必须经过的 6 个能力层：

1. **理解层**：需求解析、目标澄清、约束识别。
2. **编排层**：角色分工、计划生成、里程碑设置。
3. **执行层**：自动调用工具与工作流持续推进。
4. **治理层**：监督、干预、控制、审批、风控。
5. **交互层**：阶段汇报、状态问答、策略改写。
6. **交付层**：输出成品、质量说明、成本说明、复盘建议。

任一层缺失，都不应称为“全智化数字公司执行系统”。

### 8.5 “减少造轮子”落地清单

- 保留：LangGraph（编排）、APScheduler（调度）、Casbin（权限）、LiteLLM（网关）。
- 不自研：基础向量库、基础日志系统、基础队列中间件。
- 自研重点：
  1) 角色协作协议（Role Contract）
  2) 任务治理协议（Governance Contract）
  3) 交付物标准（Delivery Contract）

## 9. 更新后的执行建议（面向 PRD v0.2）

1. 在 PRD 首页新增“产品核心思想”章节：
   - 用户提需求，数字化公司自洽协作执行。
   - 全程可监督、可干预、可控制、可交互、可无值守。
2. 将 V1-MVP 的“技术功能”改写为“能力闭环验收项”：
   - 目标理解准确度、协作稳定度、干预响应时延、交付可用率。
3. 新增“竞品对照表”：
   - 对比维度从功能数量改为“组织执行能力与治理能力”。
4. 所有新增模块需回答一个问题：
   - 是否增强“数字化公司执行力”？若否，延后。
